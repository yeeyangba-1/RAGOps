# RAGOps 开发路线

## 1. 路线原则

- 先确认用户场景、质量目标和验收标准，再开发功能。
- 先建立版本化数据契约和可复现基线，再建设页面和自动优化。
- 每个阶段都必须有测试、文档和退出条件。
- 路线按能力与门禁排序，不在需求和团队容量未知时承诺日期。
- Sprint 0 只完成审计与设计，不包含业务代码。

## 2. Sprint 0：原型审计与方向设计（当前）

### 目标

理解 StudyRAG 与 SearchInsight 的真实代码边界，确定 RAGOps 不复制原型的整合方向。

### 交付物

- `docs/CODE_AUDIT.md`：两个项目的功能、结构、流程、复用点、问题与比较。
- `docs/ARCHITECTURE.md`：项目定位、场景、逻辑架构、模块和数据模型方向。
- `docs/ROADMAP.md`：分阶段开发路线和质量门禁。

### 退出条件

- 所有关键结论能追溯到固定代码提交与具体文件。
- 明确“不复制 StudyRAG 单体”和“不复制 SearchInsight 八 Agent 展示”的边界。
- 不存在业务代码、依赖或原项目修改。

## 3. Sprint 1：需求基线与工程地基

### 目标

把 Sprint 0 的架构方向转化为可验收的需求和稳定工程契约。

### 计划

1. 选定首个目标用户、知识库场景和部署边界。
2. 定义 MVP 用例、非功能需求、验收标准和明确非目标。
3. 编写 ADR：应用形态、存储、任务执行、provider、权限与数据保留。
4. 定义并评审首版领域 schema：版本、trace、dataset、evaluation、issue、experiment。
5. 定义错误模型、任务状态、幂等和 schema 演进规则。
6. 建立包结构、配置管理、静态检查、单元测试和 CI。
7. 准备最小人工标注评测集与数据使用规范。

### 退出条件

- 每个 MVP 用例都有输入、输出、异常和验收测试描述。
- trace 与 evaluation schema 通过示例数据和契约测试。
- CI 能执行格式、静态检查、单元测试和安全/依赖基础检查。
- 尚未明确的事项不以默认技术实现掩盖。

## 4. Sprint 2：可追溯 RAG 最小纵切

> 仅在 Sprint 1 确认 MVP 需要内置 RAG runtime 后执行；若决定先接外部系统，则改为 trace ingestion 纵切。

### 目标

完成一个端到端、可追溯、可替换 provider 的最小运行链路。

### 计划

- 文档版本、解析产物、chunk provenance 和索引版本。
- 检索接口与 `RetrievalHit`，保存 rank、分数和来源定位。
- 生成接口、Prompt/模型配置版本和 citation。
- `QueryRun`、`RetrievalRun`、`GenerationRun` 的完整 trace。
- 最小 API/CLI 调试入口；页面不是本阶段成功的必要条件。
- parser、retriever、generator 的单元、契约和端到端测试。

### 退出条件

- 任一回答可通过 `trace_id` 还原文档/索引、检索证据、Prompt、模型、耗时和状态。
- 重启后元数据与索引版本仍可恢复。
- provider 替换不影响领域用例测试。
- 失败和超时不会生成“看似成功”的不完整 trace。

## 5. Sprint 3：离线评测 MVP

### 目标

建立可复现的评测运行，替代固定分数阈值和硬编码 Bad Case 规则。

### 计划

- 评测数据集、样本、标签和不可变版本。
- evaluator registry 与 `MetricResult`。
- 首批检索指标、生成指标、系统指标和拒答评测。
- 可选 LLM Judge adapter，记录 rubric、Prompt、模型、成本和降级状态。
- 人工复核入口与 evaluator 校准流程。
- 基线配置的评测报告和可重复执行命令。

### 退出条件

- 同一代码、配置和数据集版本可重现同一确定性指标。
- LLM Judge 结果与人工标签完成小样本一致性评估。
- `failed/skipped/fallback` 不计入正常成功指标。
- 指标阈值有数据依据并记录决策，而不是沿用原型常量。

## 6. Sprint 4：质量分析与问题闭环

### 目标

把原子评测结果转化为可操作、可追踪的质量问题。

### 计划

- 多标签 issue taxonomy：检索、知识、上下文构造、生成、引用、拒答、性能等。
- 按意图、数据集标签、知识来源、配置版本和时间切片。
- Bad Case 证据视图与人工确认/驳回。
- Issue 负责人、状态、原因和修复建议。
- 质量趋势与回归视图。

### 退出条件

- 每个 Issue 能追溯到 trace、评测结果、evaluator 和原始证据。
- 同一样本可表达多个问题，不受单标签优先级遮蔽。
- 建议不会自动修改生产系统，必须经人工确认进入实验。

## 7. Sprint 5：实验对比与发布门禁

### 目标

验证优化是否真实提升质量，并阻止回归。

### 计划

- Prompt、chunk、embedding、retrieval、rerank 和模型配置变体。
- 基线/候选在同一数据集快照上的对比运行。
- 分指标、分切片差异和成本/延迟权衡。
- 版本化 gate policy 与可审计决策。
- CI/CLI 回归门禁；必要时支持人工批准。

### 退出条件

- 每次候选发布都有基线、数据集、指标差异和门禁结果。
- 提升结论同时报告退化切片、成本和延迟。
- 发布和回滚指向不可变 `RagConfigVersion`。

## 8. Sprint 6：在线观测与运营化

### 目标

连接线上 trace 与离线评测，形成长期质量趋势。

### 计划

- 在线 trace 接入、采样、脱敏和保留策略。
- 显式反馈、隐式行为与人工标签的独立采集。
- 质量、成功率、延迟、token 和成本监控。
- 数据漂移、知识更新和异常切片告警。
- 在线 Bad Case 回流评测集的审核流程。

### 退出条件

- 在线指标可按配置和索引版本解释变化。
- 敏感数据处理、权限和审计通过安全评审。
- 线上样本进入评测集前经过治理，不直接污染基准集。

## 9. Sprint 7：可靠性与规模化

### 目标

在真实负载和组织使用下保证安全、稳定与可维护。

### 计划

- 后台任务重试、取消、恢复、幂等和死信处理。
- workspace 隔离、RBAC、secret 管理和审计日志。
- 存储生命周期、归档、备份和灾难恢复。
- 性能、容量、故障注入和成本测试。
- 公共 API/SDK 兼容策略与运维手册。

### 退出条件

- 达成 Sprint 1 定义的 SLO、容量和恢复目标。
- 关键故障路径有演练记录和自动化测试。
- schema/API 变更有兼容与迁移方案。

## 10. 贯穿各阶段的质量门禁

任何实现型 Sprint 都必须满足：

- 功能开始前明确需求、范围和验收标准。
- 在独立分支开发，不直接修改 `main`。
- 修改代码必须补充或更新测试。
- 重要架构与技术选择记录 ADR。
- 数据 schema、API 和 evaluator 行为有契约测试。
- 变更通过相关单元、集成和端到端测试。
- 文档与实际实现同步更新。
- 未经验证的模型建议不得自动作用于生产配置。

## 11. Sprint 1 前必须确认的问题

1. RAGOps 首个真实使用场景和用户是谁？
2. 首版是内置 RAG runtime，还是优先接入现有 RAG trace？
3. 第一批真实/脱敏日志和人工标注数据是否可用？
4. 首版必须支持哪些文件类型、语言、数据量和并发？
5. 部署环境、第三方模型使用限制和数据安全要求是什么？
6. 哪些指标决定 MVP 成功，哪些指标可以作为发布门禁？

这些问题确认后，再把 Sprint 1 拆成具体 issue 和验收任务。
